# Writing Requirements

## Content Depth
- Target 2500-3000 words
- Reference Flutter/Dart source code where relevant (use actual file names like `framework.dart`, `widgets/framework.dart`)
- Include at least 3 substantial code examples
- Explain implementation details, not just API usage
- Discuss performance implications and memory considerations
- Mention edge cases and gotchas from real-world usage

## Code Examples
- Use realistic scenarios, not trivial examples
- Show before/after comparisons where relevant
- Include inline comments explaining non-obvious parts
- Prefer complete, runnable snippets over fragments
- Test edge cases in examples

## Structure
1. **Hook** (150-200 words): Start with a specific problem or misconception
2. **Deep Dive** (1500-1800 words): The meat - explain internals, show source references
3. **Practical Examples** (600-800 words): Real-world application
4. **Trade-offs** (200-300 words): When to use, when not to use, alternatives
5. **Key Takeaway** (100-150 words): One memorable insight

## Tone & Style
- Direct and opinionated (like a tech lead reviewing code)
- Use "you" not "we" (avoid "in this article we will...")
- Include personal insights: "I've found...", "In production, this breaks when..."
- Vary sentence length: mix short punchy statements with detailed explanations
- Use contractions (don't, it's, you've)
- Skip generic intros about Flutter being cross-platform

## Technical Accuracy
- Specify Flutter/Dart versions when behavior differs (e.g., "In Flutter 3.10+...")
- Link to actual Flutter GitHub code when referencing internals
- Mention breaking changes if relevant
- Use correct terminology (RenderObject not "render widget", BuildContext not "context object")

## What to AVOID
- Phrases like "in conclusion", "to sum up", "in this article"
- Obvious statements ("Flutter is a popular framework")
- Placeholder code or TODO comments
- Excessive use of "simply", "just", "obviously"
- Apologetic language ("you might want to consider maybe...")
- Any mention of AI assistance or generation

# Output Format
- Pure Markdown
- Use ## for main sections, ### for subsections
- Code blocks with ```dart language specifier
- No front-matter or metadata
- Ready to paste into Medium's import tool

Write at the level of:
